# Session: cc-hud-skill-tracking-ac3-planning

## Goal
Add skill/agent/read utilization tracking to cc-hud ingest pipeline, improve chart tooltips and UI panels, and begin planning an AC3 (Armored Core 3) inspired aesthetic HUD mode.

## Current State
All coding work committed across 3 commits on main (ahead of origin by 3):
- `eb24af7` Combined cost chart: persistent total label, running total in tooltip
- `5167e82` Skill/read extraction, scrollable tool panel, snapshot tests, hover fixes
- `036cb2d` Move tests into source file, delete lib.rs and tests/ dir

## Problem/Context
cc-hud ingested JSONL session data but only tracked raw tool names. Skill invocations showed up as generic "Skill" tool_use counts. Agent spawns had no metadata. No way to see what skills were loaded, which subagent types were used, or when CLAUDE.md/memory files were read.

## Solution/Approach

### Ingest changes (claude_code.rs)
- ContentBlock now parses `input` field from tool_use blocks
- New Event variants: `SkillUse { skill }`, `ReadFile { category }`, extended `AgentSpawn { subagent_type, description }`
- Skill name extracted from `input.skill`, agent type from `input.subagent_type`
- Read tool classified by file_path: "CLAUDE.md" or "memory" categories
- `skill_counts` and `read_counts` added to SessionData and SubagentData

### UI changes (main.rs)
- Tool panel converted from raw painter to ScrollArea with draw_bar_row helper
- Three scrollable sections: tool calls, skills (namespace-stripped), reads
- Skill marker vlines (green, SKILL_MARKER color) on cost/turn, total cost, strip charts
- Combined cost chart shows persistent total in top-right label
- Hover tooltip: removed snap distance (shows at any x position), added running total header for WeeklyCost, shows nearby skill names
- Skill legend entry in strip view

### Tests
- insta snapshot tests colocated in claude_code.rs (#[cfg(test)] mod tests)
- Fixture at src/agent_harnesses/fixtures/sample_session.jsonl covering all event types
- 3 tests: events snapshot, session aggregates snapshot, event type coverage

## Key Insights
- Sampler (sqshq/sampler, Go) has clean component architecture worth porting: grid layout, consumer channels, minimal component interface. ~2-3k lines Rust to replicate core
- AC3 HUD element mapping: AP=tokens, Energy=keystroke rate, Heat=CPU temp, Radar=mouse position, Weapon slots=recent tools, Ammo=call counts
- Braille rendering better done as egui Painter circles/rects faking the dot-grid aesthetic rather than actual unicode braille characters -- avoids cell alignment issues with box drawing
- Chrome rendering pipeline (dirty rects, layer compositing, tile raster) studied for context on egui optimization. Render-to-texture via PaintCallback is the applicable optimization for static aesthetic layers
- User strongly prefers colocated tests inside source files, not separate tests/ directory
- User does not want relevance judgements ("this doesn't matter for your case")

## Open Questions
- AC3 mode: separate binary or mode toggle within cc-hud?
- Component sizing: aspect ratio declaration vs explicit grid placement?
- Data interface: normalized 0-1 scalar gauge vs time series for sparkline types?
- Which data sources beyond JSONL: mouse position, keystroke rate, CPU temp, audio?
- How dirty aesthetically: scanline intensity, jitter, chromatic aberration, phosphor glow levels?
