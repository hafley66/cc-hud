# Session: cc-hud-time-mode-plot-bounds-fix

## Goal
Fix time-mode rendering: charts appeared blank when toggling to time axis.

## Current State
FIXED. Compiles clean, user confirmed time mode renders.

## Problem/Context
egui_plot persists bounds in memory keyed by plot ID. All five charts shared an ID across modes ("cost_big", "tok_big", "energy_wh", "water_ml", "totals_combined"). Toggling turn→time kept the turn-mode X bounds (~0..200) even though data X values were minutes-from-epoch (~29.5M). Data rendered entirely off-screen.

Builder methods `include_x` + `auto_bounds([false, true])` only seed `min_auto_bounds` (the reset value); they do NOT overwrite the live memorized bounds once the plot has been shown once.

Debug trace (eprintln) confirmed: active_view spanned data correctly, cost_rect was valid, lines/bars populated with correct X ranges. The plot itself was ignoring the view.

## Solution/Approach
Inside each plot's `show(ui, |pui| { ... })` closure, added as the first two lines:
```
pui.set_plot_bounds_x(vis_xmin..=vis_xmax);
pui.set_auto_bounds([false, true]);
```
`vis_xmin`/`vis_xmax` are already computed once per frame at src/main.rs:1679 from `active_view` (falls back to full range when None). Applied via `Edit replace_all` on the literal `let plot_resp = p.show(ui, |pui| {` — hit all 7 occurrences in one edit.

## Key Insights
- `Plot::include_x` only extends `min_auto_bounds` (reset seed). With `default_auto_bounds.x = false`, it has no effect on current bounds after the first render.
- `PlotUi::set_plot_bounds_x(range)` is the correct API for programmatically driving X bounds each frame while leaving Y auto.
- `Plot::default_x_bounds(min, max)` also sets `default_auto_bounds.x = false` — useful for initial bounds, not per-frame updates.
- Plot memory is keyed by the `Plot::new(id)` string. Changing the ID per mode is an alternative (cheaper) fix but loses pan/zoom state across toggles.
