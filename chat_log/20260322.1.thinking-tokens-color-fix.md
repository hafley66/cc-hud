Applied across scene builder, all 4 egui chart instances, and tooltip meter.

### Thinking token investigation
Discovered Claude Code v2.1.81 writes multiple JSONL entries per API call:
- Thinking partial: stop_reason=null, stale/intermediate output_tokens (8-14)
- Text partial: stop_reason=null, stale output_tokens
- Final entry: stop_reason!=null, real cumulative output_tokens

The thinking-only entries' tiny output_tokens are NOT the thinking token count.
They're stale streaming counters. Real output_tokens (including thinking) are
only on the final entry. Thinking tokens are lumped into output_tokens with no
separate field in the API response (confirmed via web search of API docs).

### Streaming partial merge (ingest fix)
Rewrote tail_jsonl to accumulate partials by message ID:
- Added `id` and `stop_reason` fields to JsonlMessage
- Track pending state per message ID (has_thinking, tool_events, timestamp)
- Only emit ApiCall event on final entry (stop_reason != null)
- Tool/skill/agent events accumulated from all partials
- Eliminates duplicate turns and uses real usage numbers

### mitmproxy addon (proxy/thinking_capture.py)
Stream-tapping proxy for Claude API. Copies SSE chunks into buffer while
forwarding unmodified. Parses thinking_delta events on completion. Not needed
for token counts (they're in the JSONL) but could capture thinking text
content which is redacted via redact-thinking-2026-02-12 beta flag.

## Key Findings
- Claude Code binary: Mach-O arm64 with bundled minified JS, NOT plain Node
- Beta flag `redact-thinking-2026-02-12` causes API to return empty thinking text
- `output_tokens` includes thinking tokens, no separate field exists
- Streaming partials have stale/intermediate usage, only final entry is real
- Thinking and content always split into separate JSONL entries (same msg ID)
- The 8-27 token "thinking" entries were misleading -- just streaming overhead
